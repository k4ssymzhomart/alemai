# official_sources_notes — Нормативный радар (R14–R21)

> Аудит 2026-07-09, четыре параллельных зонда (web_fetch + web-поиск; bash/curl не использовались). Задача — НЕ скачивание текстов (корпус в `regs/`), а поиск стабильных сигналов версии и ссылок-целей. Companion: `official_sources.csv` (34 строки × 12 колонок, провалидирован python csv). Конфиденс: CONFIRMED = наблюдалось сегодня; LIKELY = вывод из наблюдений соседних URL/вчерашней сессии; UNKNOWN = не наблюдаемо.

## 0. Сквозная анатомия зеркала zakon.uchet.kz (главный рабочий источник радара)

Все страницы актов на зеркале собраны по одному шаблону; слот маркера версии одинаков:

```
[шапка сайта, декой-даты]  →  «Оглавление»  →  H1 (название акта)
→ H4-строка реквизитов «Приказ/Постановление … № NNNNN.» (заканчивается рег. номером)
→ ОТДЕЛЬНЫЙ АБЗАЦ-МАРКЕР: «Обновленный с изменениями на: DD.MM.YYYY»
   (либо, для ни разу не менявшихся актов, статус-слово «Действующий»)
→ [иногда «Примечание ИЗПИ! …»] → преамбула → текст акта
→ (только на коротких страницах) инфо-таблица «Дата изменения акта | DD.MM.YYYY»,
  «Статус | Обновленный», таблица «История изменения документа»
  со ссылками /rus/history/<DOCID>/DD.MM.YYYY
```

Наблюдения, критичные для парсера:

1. **Маркер всегда в первых ~3–7% текста** (наблюдался на строках 52–58 из ~720–760, либо в первых ~3–5 тыс. знаков) — переживает любую обрезку.
2. **Обрезка длинных страниц на ~57–86 тыс. знаков** (21550 @56 980; 21904-kaz @67 059; 21831 @68 450; 21744 @86 234; P2500000840 @82 155). Хвостовые якоря (история, инфо-таблица) на длинных актах НЕДОСТИЖИМЫ — единственный надежный сигнал — топ-маркер.
3. **Декой-даты в шапке**: «Состояние базы… Дата обновления: 16.01.2024» стоит ВЫШЕ маркера на каждой странице. Никогда не брать «первую дату на странице» — только совпадение по фразе маркера.
4. **Недетерминированный рендер**: 09.07 страница 21831 пришла ОДНОЙ строкой в 68 тыс. знаков (6 строк вывода всего), 21550 — «схлопнутыми» строками, остальные — построчно. Regex обязан быть newline-agnostic: без `^`/`$`, без предположения «маркер = отдельная строка».
5. **kk-страницы**: 09.07 маркер имел вид `с изменениями на: 27.02.2026` (БЕЗ префикса); 08.07 наблюдалась форма `Өзгерістерімен жаңартылған күні: …`. Формат плавает — матчить обе. На kk-страницах инфо-таблицы подписаны по-русски («Дата изменения акта | 27.02.2026»).
6. **Первичные редакции** (акт ни разу не менялся — ПП 840, ПП 1086): маркера с датой НЕТ, в его слоте отдельная строка-статус «Действующий». Появление «Обновленный с изменениями на:» на таких страницах = первая правка.
7. Куки-воллов, капчи, редиректов нет (redirect URL = тот же URL); мусор — GTM-iframe и счетчик mail.ru. Кодировка чистая UTF-8. H1/meta-титулы обрезаются на ~250 знаках посреди слова — не использовать их для сверки названий.

### Рекомендуемая regex-библиотека

| Назначение | Паттерн |
|---|---|
| Универсальный маркер ru+kk (первое вхождение) | `(?:Обновленный\s+)?с изменениями на:\s*(\d{2}\.\d{2}\.\d{4})` |
| Легаси-альтернатива kk (на всякий случай) | `Өзгерістерімен жаңартылған күні:\s*(\d{2}\.\d{2}\.\d{4})` |
| Фолбэк (короткие страницы, низ) | `Дата изменения акта\D{0,10}(\d{2}\.\d{2}\.\d{4})` |
| Статус-слово первичной редакции | отдельное вхождение `Действующий` сразу после H4-строки реквизитов «… № NNN» |

Сравнение — как **непрозрачный токен**: маркер иногда равен дате приказа-изменителя (21550: 31.12.2025), иногда дате вступления правок (21263: 01.01.2026). Семантику не интерпретировать; «изменился → показать пользователю + quick-link».
